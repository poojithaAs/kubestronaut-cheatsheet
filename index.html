Chapter 1: Kubernetes Architecture

What is Kubernetes?

Kubernetes (K8s) is a container orchestration platform. Think of it as the operating system for your data center. Instead of manually placing containers on servers, you tell Kubernetes what you want (3 replicas of my app, 512MB each) and it figures out where and how.

In your org: Without K8s, your team SSHs into servers, pulls Docker images, manages ports, restarts crashed containers manually. With K8s, you write a YAML file, apply it, and K8s handles placement, scaling, healing, and networking across hundreds of nodes automatically.

Control Plane Components

┌─────────────────── CONTROL PLANE ────────────────────┐ │ │ │ ┌──────────┐ ┌───────────┐ ┌────────────────────┐│ │ │ API │ │ etcd │ │ Controller Manager ││ │ │ Server │◄─│ (key-val │ │ (watches desired ││ │ │ (gateway │ │ store) │ │ vs actual state) ││ │ │ for ALL │ └───────────┘ └────────────────────┘│ │ │ requests│ │ │ └────┬─────┘ ┌───────────────────────────────────┐│ │ │ │ Scheduler (assigns pods to nodes) ││ │ │ └───────────────────────────────────┘│ └───────┼──────────────────────────────────────────────┘ │ ┌───────▼──────────── WORKER NODES ────────────────────┐ │ ┌─────────┐ ┌────────────┐ ┌──────────────────┐ │ │ │ kubelet │ │ kube-proxy │ │ Container Runtime│ │ │ │ (agent) │ │ (network │ │ (containerd/CRI) │ │ │ └─────────┘ │ rules) │ └──────────────────┘ │ │ └────────────┘ │ │ ┌────┐ ┌────┐ ┌────┐ ┌────┐ │ │ │Pod │ │Pod │ │Pod │ │Pod │ │ │ └────┘ └────┘ └────┘ └────┘ │ └──────────────────────────────────────────────────────┘

API Server: The front door. Every kubectl command, every controller, every kubelet talks to the API server. It validates requests and writes state to etcd.

etcd: The database. Stores the entire cluster state — every pod, service, secret, configmap. If etcd dies, the cluster loses its memory.

Scheduler: Decides which node runs a new pod. Considers resource requests, affinity rules, taints/tolerations. Think of it as the air traffic controller.

Controller Manager: Runs control loops. The Deployment controller ensures 3 replicas exist. The Node controller marks nodes unhealthy. Each controller watches desired state vs actual state and reconciles.

kubelet: The agent on every worker node. Receives pod specs from the API server, tells the container runtime to start containers, reports health back.

kube-proxy: Handles networking rules on each node so services can route traffic to the right pods.

Core Objects

Pod: The smallest deployable unit. One or more containers that share networking and storage. In practice, most pods have one container. Think of a pod as a wrapper around your container.

ReplicaSet: Ensures N copies of a pod are always running. If one dies, it creates a replacement. You almost never create these directly — Deployments manage them.

Deployment: The most common workload resource. Manages ReplicaSets, handles rolling updates, rollbacks. You define your desired state, the Deployment controller makes it happen.

Service: A stable network endpoint for a set of pods. Pods get random IPs and die/restart constantly. A Service gives you a DNS name (my-app.default.svc.cluster.local) that always routes to healthy pods.

Namespace: A virtual cluster within a cluster. Use namespaces to isolate teams, environments, or projects. Default namespaces: default, kube-system, kube-public, kube-node-lease.

# Create a deployment with 3 replicas kubectl create deployment nginx --image=nginx --replicas=3 # Expose it as a service kubectl expose deployment nginx --port=80 --type=ClusterIP # See everything kubectl get all -n default
Exam tip: 46% of KCNA is K8s fundamentals. Know the control plane components, what each does, and the relationship between Deployments → ReplicaSets → Pods. This is half your exam.

Chapter 2: Configuration & Organization

ConfigMaps

Store non-sensitive configuration data as key-value pairs. Your app reads database hostnames, feature flags, log levels from ConfigMaps instead of hardcoding them.

# Create a ConfigMap kubectl create configmap app-config \ --from-literal=DB_HOST=postgres.prod.svc \ --from-literal=LOG_LEVEL=info # Use in a pod as env vars env: - name: DB_HOST valueFrom: configMapKeyRef: name: app-config key: DB_HOST

Secrets

Like ConfigMaps but for sensitive data — passwords, tokens, TLS certs. Base64 encoded (NOT encrypted by default). In production, enable encryption at rest with EncryptionConfiguration or use an external secrets manager (Vault, AWS Secrets Manager).

# Create a secret kubectl create secret generic db-creds \ --from-literal=password=MyS3cr3t # Secrets are base64 encoded, NOT encrypted # Always enable encryption at rest in production
Common mistake: Base64 is encoding, not encryption. Anyone with cluster access can decode secrets. For real security: enable etcd encryption at rest + use RBAC to restrict who can read secrets + consider external secret stores.

Labels & Selectors

Labels are key-value tags attached to objects. Selectors filter objects by labels. This is how Services find their pods, how Deployments manage ReplicaSets, and how you organize everything.

# Label a pod kubectl label pod my-pod env=production team=payments # Select by label kubectl get pods -l env=production kubectl get pods -l "env=production,team=payments" kubectl get pods -l "env in (production,staging)"
In your org: Every resource gets labels: app=checkout, env=prod, team=payments, version=v2.1. Your service selects app=checkout to route traffic. Your monitoring uses team=payments to send alerts to the right Slack channel. Labels are the glue of Kubernetes.

Chapter 3: Containers & Container Runtimes

Containers 101

A container packages your app + dependencies into an isolated process. Unlike VMs (which virtualize hardware), containers share the host kernel but isolate using Linux namespaces (process, network, filesystem) and cgroups (CPU, memory limits).

VM Approach: Container Approach: ┌──────────────────┐ ┌──────────────────┐ │ App A │ App B │ │ App A │ App B │ │──────────┼───────│ │──────────┼───────│ │ Guest OS│GuestOS│ │ Container│Contner│ │──────────┼───────│ │ Runtime │Runtime│ │ Hypervisor │ │──────────────────│ │──────────────────│ │ Host OS │ │ Host OS │ │──────────────────│ │──────────────────│ │ Hardware │ │ Hardware │ └──────────────────┘ └──────────────────┘ Heavy, slow boot, GB size Light, instant, MB size

Container Runtime Interface (CRI)

Kubernetes doesn't run containers directly. It talks to a container runtime through the CRI standard. The default runtime is containerd. Docker was removed as a runtime in K8s 1.24 (but Docker-built images still work fine — images are OCI standard).

Scheduling

The scheduler places pods on nodes based on: resource requests/limits, node selectors, affinity/anti-affinity rules, taints/tolerations, and topology spread constraints.

# Node selector: run only on SSD nodes spec: nodeSelector: disk: ssd # Taint a node (only tolerated pods can run here) kubectl taint nodes gpu-node-1 gpu=true:NoSchedule # Toleration in pod spec tolerations: - key: "gpu" operator: "Equal" value: "true" effect: "NoSchedule"
In your org: You taint GPU nodes so only ML workloads run there. You use pod anti-affinity to spread your 3 database replicas across 3 availability zones. You set resource requests so the scheduler never overcommits a node's memory.

Networking Basics

K8s networking rules: every pod gets its own IP. Pods on any node can communicate with any other pod without NAT. Services provide stable IPs and DNS names. NetworkPolicies act as firewalls between pods.

Storage Basics

Containers are ephemeral — data is lost when the container restarts. For persistent data: PersistentVolumes (PV) represent storage, PersistentVolumeClaims (PVC) request storage, StorageClasses enable dynamic provisioning.

# Dynamic provisioning: just create a PVC apiVersion: v1 kind: PersistentVolumeClaim metadata: name: db-data spec: accessModes: [ReadWriteOnce] storageClassName: gp3 resources: requests: storage: 50Gi # K8s automatically creates the underlying EBS/disk

Chapter 4: Cloud Native Architecture & CNCF

What is Cloud Native?

Cloud native = building apps that fully exploit cloud advantages: containers for packaging, orchestration for management, microservices for modularity, CI/CD for speed, observability for insight. It's not just "running in the cloud" — it's designing for the cloud.

Microservices vs Monolith

Monolith: One codebase, one deployment, one database. Simple to start, hard to scale individual components. A bug in the checkout module can crash the entire app.

Microservices: Each feature is an independent service with its own database, deployed independently. The checkout service can scale to 10 replicas during Black Friday while the profile service stays at 2.

In your org: You start with a monolith (fast to build). When it gets too big to deploy safely, you extract the highest-traffic components into microservices first. Not everything needs to be a microservice — extract when you have a reason (scaling, independent deployment, different tech stack).

Serverless

Functions that run on demand. No servers to manage. You pay per execution. AWS Lambda, Google Cloud Functions, Knative (on K8s). Great for event-driven tasks: image processing, webhooks, scheduled jobs. Bad for long-running processes or anything needing persistent connections.

The CNCF Landscape

The Cloud Native Computing Foundation (CNCF) hosts 170+ projects. Know these key graduated projects for the exam:

  • Kubernetes — container orchestration
  • Prometheus — metrics and alerting
  • Envoy — service proxy (used by Istio)
  • Helm — package manager for K8s
  • containerd — container runtime
  • Argo — GitOps and workflows
  • Flux — GitOps continuous delivery
  • OpenTelemetry — observability framework

Service Mesh

A dedicated infrastructure layer for service-to-service communication. Handles mTLS encryption, traffic routing, retries, circuit breaking — without changing your app code. Istio and Linkerd are the two main choices. The sidecar proxy (Envoy) sits next to each pod and handles all network traffic.

Without Service Mesh: With Service Mesh (Istio): ┌───────┐ ┌───────┐ ┌───────────────┐ ┌───────────────┐ │ App A ├──▶│ App B │ │ App A │ Envoy ├──▶│ Envoy │ App B │ └───────┘ └───────┘ └───────┼───────┘ └───────┼───────┘ (plain HTTP, no encryption, │ mTLS, retries, │ no retries, no observability) │ tracing, metrics │ └──── Control Plane ┘

Chapter 5: Observability

The Three Pillars

Logs: Event records. "User 123 logged in at 14:02." Structured JSON logs are searchable; plain text logs are not. Tools: Fluentd/Fluent Bit → Elasticsearch/Loki.

Metrics: Numeric measurements over time. "CPU usage is 73%." "Request latency p99 is 200ms." Tools: Prometheus for collection, Grafana for visualization.

Traces: Follow a single request across multiple services. "This API call took 340ms — 200ms in the payment service, 100ms in the database, 40ms in the gateway." Tools: OpenTelemetry, Jaeger, Zipkin.

In your org: Your monitoring stack: Prometheus scrapes metrics from every pod → Grafana dashboards show real-time health → Alertmanager pages your on-call when p99 latency exceeds 500ms → Loki stores logs → Jaeger shows traces. OpenTelemetry is the standard SDK that instruments your apps to emit all three.

Chapter 6: Application Delivery & GitOps

CI/CD Pipeline

Continuous Integration (CI): Every code push triggers: build → test → scan → create container image → push to registry.

Continuous Delivery (CD): After CI passes, automatically deploy to staging. Manual approval for production. Continuous Deployment = auto-deploy to production too.

GitOps

Git is the single source of truth for infrastructure and app state. Instead of running kubectl apply manually, you commit YAML to Git, and a GitOps operator (ArgoCD or Flux) syncs the cluster to match. If someone makes a manual change, the operator reverts it.

Developer pushes code │ ▼ CI Pipeline (GitHub Actions / GitLab CI) ├── Build container image ├── Run tests ├── Push image to registry (tag: v2.3.1) └── Update image tag in GitOps repo │ ▼ GitOps Repo (desired state in Git) │ ▼ ArgoCD / Flux (watches Git, syncs to cluster) │ ▼ Kubernetes Cluster (actual state matches Git)

Helm

The package manager for Kubernetes. A Helm chart is a templated bundle of K8s manifests. Instead of copying 20 YAML files for each environment, you use one chart with variables.

# Install a chart helm install my-app ./my-chart --values production.yaml # Upgrade helm upgrade my-app ./my-chart --values production.yaml # Rollback helm rollback my-app 1
Exam tip: Know the difference between CI and CD, what GitOps is, the purpose of Helm, and the names of key tools (ArgoCD, Flux, Helm, Kustomize). You don't need to know syntax — this is a conceptual exam.

SECURITY FUNDAMENTALS

Chapter 1: The 4Cs of Cloud Native Security

Security works in layers — each layer protects the one inside it. If the outer layer is breached, inner layers must still hold.

┌──────────────────────────────────────────────┐ │ CLOUD │ │ (AWS/GCP/Azure security: IAM, VPC, SGs) │ │ ┌──────────────────────────────────────────┐│ │ │ CLUSTER ││ │ │ (K8s: RBAC, NetworkPolicy, PodSecurity) ││ │ │ ┌──────────────────────────────────────┐││ │ │ │ CONTAINER │││ │ │ │ (Image scanning, read-only FS, │││ │ │ │ non-root, minimal base image) │││ │ │ │ ┌──────────────────────────────────┐│││ │ │ │ │ CODE ││││ │ │ │ │ (Input validation, SAST, deps) ││││ │ │ │ └──────────────────────────────────┘│││ │ │ └──────────────────────────────────────┘││ │ └──────────────────────────────────────────┘│ └──────────────────────────────────────────────┘
In your org: Cloud = your AWS account has VPCs, security groups, IAM roles. Cluster = RBAC limits who can do what in K8s. Container = your Dockerfile runs as non-root with a minimal base image. Code = your app validates all input and your CI runs SAST scanners. A breach at the cloud layer doesn't mean game over IF your cluster, container, and code layers are solid.
KUBERNETES CLUSTER SECURITY

Chapter 2: RBAC (Role-Based Access Control)

RBAC controls who can do what in your cluster. Four objects work together:

Role: A set of permissions (get, list, create, delete) on resources (pods, services, secrets) within a namespace.

ClusterRole: Same as Role but cluster-wide (all namespaces + non-namespaced resources like nodes).

RoleBinding: Connects a Role to a user/group/service account within a namespace.

ClusterRoleBinding: Connects a ClusterRole to a user/group cluster-wide.

# Role: allow read-only access to pods in "production" namespace apiVersion: rbac.authorization.k8s.io/v1 kind: Role metadata: namespace: production name: pod-reader rules: - apiGroups: [""] resources: ["pods"] verbs: ["get", "list", "watch"] --- # RoleBinding: give "dev-team" this role apiVersion: rbac.authorization.k8s.io/v1 kind: RoleBinding metadata: name: dev-team-pod-reader namespace: production subjects: - kind: Group name: dev-team roleRef: kind: Role name: pod-reader apiGroup: rbac.authorization.k8s.io
Principle of least privilege: Give only the permissions needed, nothing more. Developers get read access to production, write access to staging. CI/CD service accounts get deploy permissions only to their namespace. Nobody gets cluster-admin except the platform team — and even then, use it sparingly.
KUBERNETES CLUSTER SECURITY

Chapter 3: Network Policies & Pod Security

Network Policies

By default, all pods can talk to all pods. Network Policies are firewalls between pods. They use label selectors to define allowed ingress (incoming) and egress (outgoing) traffic.

# Deny all ingress to pods in "production" namespace apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: deny-all-ingress namespace: production spec: podSelector: {} # all pods in namespace policyTypes: [Ingress] # no ingress rules = deny all incoming traffic --- # Allow only frontend to talk to backend on port 8080 apiVersion: networking.k8s.io/v1 kind: NetworkPolicy metadata: name: allow-frontend-to-backend namespace: production spec: podSelector: matchLabels: app: backend ingress: - from: - podSelector: matchLabels: app: frontend ports: - port: 8080

Pod Security Standards (PSS)

Three levels that restrict what pods can do:

  • Privileged: No restrictions. Used for system-level pods (CNI plugins, monitoring agents).
  • Baseline: Blocks known privilege escalations. Prevents hostNetwork, hostPID, privileged containers. Good default for most workloads.
  • Restricted: Maximum security. Must run as non-root, read-only root filesystem, drop all capabilities. Use for sensitive workloads.
# Enforce restricted security on a namespace kubectl label namespace production \ pod-security.kubernetes.io/enforce=restricted \ pod-security.kubernetes.io/warn=restricted \ pod-security.kubernetes.io/audit=restricted
In your org: Start with baseline enforcement on all namespaces. Warn on restricted so devs see what they'd need to fix. Exempt kube-system (system pods need elevated privileges). Gradually move to restricted for production workloads.
PLATFORM SECURITY

Chapter 4: Supply Chain & Image Security

Image Security Best Practices

  • Use minimal base images: distroless or alpine. Fewer packages = fewer vulnerabilities.
  • Scan images in CI: Trivy, Grype, or Snyk scan for CVEs before pushing to registry.
  • Sign images: Cosign/Sigstore signs images so you can verify they haven't been tampered with.
  • Use image digests, not tags: nginx@sha256:abc123 is immutable. nginx:latest can change under you.
  • Private registries only: Don't pull from public Docker Hub in production. Mirror to your private registry.
# Scan image for vulnerabilities trivy image my-app:v2.1.0 # Sign image with Cosign cosign sign --key cosign.key my-registry/my-app:v2.1.0 # Verify before deploying cosign verify --key cosign.pub my-registry/my-app:v2.1.0

Admission Controllers

Gatekeepers that intercept API requests before objects are created. Use them to enforce policies: no latest tags, images must come from approved registries, pods must have resource limits.

Tools: OPA Gatekeeper, Kyverno (simpler, K8s-native policies).

Chapter 5: Compliance & Threat Modeling

Key Frameworks

  • CIS Benchmarks: Checklist of security configurations for K8s. Run kube-bench to audit your cluster against CIS standards automatically.
  • NIST: Framework for managing cybersecurity risk (Identify, Protect, Detect, Respond, Recover).
  • SOC 2: Audit standard for cloud services. Your K8s security practices feed into SOC 2 compliance.
  • PCI DSS: Payment card data security. If your cluster processes payments, network segmentation, encryption, and access controls are mandatory.

Threat Modeling: STRIDE

  • Spoofing — Fake identity → Fix: strong authentication (mTLS, OIDC)
  • Tampering — Modified data → Fix: image signing, admission controllers
  • Repudiation — Deny actions → Fix: audit logging
  • Information Disclosure — Data leak → Fix: encryption, RBAC, secrets management
  • Denial of Service — Resource exhaustion → Fix: resource limits, rate limiting
  • Elevation of Privilege — Gain admin access → Fix: least privilege RBAC, PSS
# Run CIS benchmark audit kube-bench run --targets master,node # Example output: # [PASS] 1.1.1 Ensure API server pod spec permissions are set to 600 # [FAIL] 1.2.6 Ensure --kubelet-certificate-authority is set # [WARN] 1.2.16 Ensure admission control plugin PodSecurity is set

Chapter 1: Cluster Setup with kubeadm

Installing a Cluster from Scratch

kubeadm is the official tool to bootstrap a K8s cluster. It handles certificate generation, etcd setup, control plane components, and joining worker nodes.

# On the CONTROL PLANE node: # 1. Initialize the cluster sudo kubeadm init --pod-network-cidr=10.244.0.0/16 # 2. Set up kubectl access mkdir -p $HOME/.kube sudo cp /etc/kubernetes/admin.conf $HOME/.kube/config # 3. Install a CNI (network plugin) — Calico example kubectl apply -f https://docs.projectcalico.org/manifests/calico.yaml # 4. Get the join command for worker nodes kubeadm token create --print-join-command
# On each WORKER NODE: sudo kubeadm join 192.168.1.10:6443 --token abc123 \ --discovery-token-ca-cert-hash sha256:xyz789

etcd Backup & Restore

# Backup etcd ETCDCTL_API=3 etcdctl snapshot save /tmp/etcd-backup.db \ --endpoints=https://127.0.0.1:2379 \ --cacert=/etc/kubernetes/pki/etcd/ca.crt \ --cert=/etc/kubernetes/pki/etcd/server.crt \ --key=/etc/kubernetes/pki/etcd/server.key # Restore etcd (disaster recovery) ETCDCTL_API=3 etcdctl snapshot restore /tmp/etcd-backup.db \ --data-dir=/var/lib/etcd-restored # Then update etcd pod manifest to point to new data dir
Exam critical: etcd backup/restore is almost guaranteed on the CKA exam. Memorize the flags: --endpoints, --cacert, --cert, --key for backup. --data-dir for restore.

Chapter 2: RBAC & Cluster Upgrades

RBAC in Practice

# Create a service account for CI/CD kubectl create serviceaccount deploy-bot -n production # Create a role with deploy permissions kubectl create role deployer --verb=get,list,create,update,patch \ --resource=deployments,services,configmaps -n production # Bind the role to the service account kubectl create rolebinding deploy-bot-binding \ --role=deployer --serviceaccount=production:deploy-bot -n production # Test permissions kubectl auth can-i create deployments -n production --as=system:serviceaccount:production:deploy-bot # yes

Cluster Upgrade (kubeadm)

Upgrade one minor version at a time (e.g., 1.34 → 1.35, never skip). Control plane first, then workers.

# Step 1: Upgrade kubeadm on control plane sudo apt-get update sudo apt-get install -y kubeadm=1.35.0-00 # Step 2: Plan the upgrade sudo kubeadm upgrade plan # Step 3: Apply the upgrade sudo kubeadm upgrade apply v1.35.0 # Step 4: Drain a worker node (evict pods safely) kubectl drain worker-1 --ignore-daemonsets --delete-emptydir-data # Step 5: Upgrade kubelet + kubectl on the worker sudo apt-get install -y kubelet=1.35.0-00 kubectl=1.35.0-00 sudo systemctl restart kubelet # Step 6: Uncordon the node (allow scheduling again) kubectl uncordon worker-1
In your org: You upgrade staging first, run your test suite, wait a week. Then upgrade production control plane during a maintenance window. Drain and upgrade workers one at a time (rolling) so there's zero downtime. PodDisruptionBudgets ensure critical services always have minimum replicas during drain.

Chapter 3: Workloads & Scheduling

Deployment Strategies

# Rolling Update (default) — gradually replace old pods spec: strategy: type: RollingUpdate rollingUpdate: maxSurge: 1 # 1 extra pod during update maxUnavailable: 0 # never drop below desired count # Recreate — kill all old pods, then create new ones # Use when you can't run two versions simultaneously (DB schema changes) spec: strategy: type: Recreate

DaemonSets, Jobs, CronJobs

DaemonSet: Runs one pod per node. Use for: log collectors (Fluentd), monitoring agents (node-exporter), CNI plugins.

Job: Run-to-completion workload. Runs a pod until it succeeds. Use for: database migrations, batch processing.

CronJob: Scheduled Job. Runs on a cron schedule. Use for: nightly backups, hourly report generation.

# CronJob: backup database every night at 2 AM apiVersion: batch/v1 kind: CronJob metadata: name: db-backup spec: schedule: "0 2 * * *" jobTemplate: spec: template: spec: containers: - name: backup image: postgres:15 command: ["pg_dump", "-h", "db-host", "-U", "admin", "mydb"] restartPolicy: OnFailure

Resource Requests & Limits

resources: requests: # Guaranteed minimum (used for scheduling) cpu: "250m" # 250 millicores = 0.25 CPU memory: "256Mi" limits: # Maximum allowed (enforced by cgroups) cpu: "500m" # Throttled if exceeded memory: "512Mi" # OOMKilled if exceeded

Requests = what the scheduler guarantees (used for placement). Limits = the ceiling. CPU over-limit = throttled. Memory over-limit = OOMKilled (exit code 137).

Taints, Tolerations & Affinity

# Taint a node: only special workloads can run here kubectl taint nodes gpu-node nvidia.com/gpu=true:NoSchedule # Pod tolerates the taint tolerations: - key: "nvidia.com/gpu" operator: "Equal" value: "true" effect: "NoSchedule" # Pod affinity: schedule near pods with label app=cache affinity: podAffinity: requiredDuringSchedulingIgnoredDuringExecution: - labelSelector: matchLabels: app: cache topologyKey: kubernetes.io/hostname # Pod anti-affinity: spread across zones affinity: podAntiAffinity: requiredDuringSchedulingIgnoredDuringExecution: - labelSelector: matchLabels: app: database topologyKey: topology.kubernetes.io/zone

Chapter 4: Services & Networking

Service Types

  • ClusterIP (default): Internal only. Pods inside the cluster can reach it. Used for service-to-service communication.
  • NodePort: Exposes on a static port (30000-32767) on every node. External traffic hits NodeIP:NodePort → routed to pods. Used for dev/testing.
  • LoadBalancer: Provisions a cloud load balancer (ELB, NLB). External traffic → LB → pods. Used for production external services.
  • ExternalName: Maps to a DNS name (CNAME). No proxying. Use when pointing to external services.
# ClusterIP (default) kubectl expose deployment backend --port=8080 --type=ClusterIP # NodePort kubectl expose deployment frontend --port=80 --type=NodePort # LoadBalancer kubectl expose deployment api --port=443 --type=LoadBalancer

Ingress

A single entry point for external HTTP/HTTPS traffic. Routes by host/path to different services. Requires an Ingress Controller (nginx, Traefik, AWS ALB).

apiVersion: networking.k8s.io/v1 kind: Ingress metadata: name: app-ingress annotations: nginx.ingress.kubernetes.io/rewrite-target: / spec: rules: - host: api.mycompany.com http: paths: - path: /v1 pathType: Prefix backend: service: name: api-v1 port: number: 80 - path: /v2 pathType: Prefix backend: service: name: api-v2 port: number: 80 tls: - hosts: - api.mycompany.com secretName: tls-secret

DNS in Kubernetes

CoreDNS runs as pods in kube-system. Every service gets a DNS record: <service>.<namespace>.svc.cluster.local. Pods in the same namespace can use just the service name. Cross-namespace: service-name.other-namespace.

# From within a pod: curl http://backend # same namespace curl http://backend.production # cross-namespace curl http://backend.production.svc.cluster.local # full FQDN

Chapter 5: Storage

PV, PVC, StorageClass

StorageClass (gp3, io2, standard) │ │ dynamically provisions ▼ PersistentVolume (PV) ◄──── bound ────► PersistentVolumeClaim (PVC) (the actual disk) (the request from a pod) │ ▼ Cloud Disk (EBS, GCE PD, Azure Disk)
# StorageClass for AWS gp3 apiVersion: storage.k8s.io/v1 kind: StorageClass metadata: name: gp3 provisioner: ebs.csi.aws.com parameters: type: gp3 reclaimPolicy: Retain # Keep disk when PVC is deleted volumeBindingMode: WaitForFirstConsumer # Bind to node's AZ --- # PVC apiVersion: v1 kind: PersistentVolumeClaim metadata: name: postgres-data spec: accessModes: [ReadWriteOnce] storageClassName: gp3 resources: requests: storage: 100Gi --- # Pod using the PVC spec: volumes: - name: data persistentVolumeClaim: claimName: postgres-data containers: - name: postgres volumeMounts: - name: data mountPath: /var/lib/postgresql/data

Access Modes: ReadWriteOnce (one node), ReadOnlyMany (many nodes read), ReadWriteMany (many nodes read/write — needs NFS or EFS).

Reclaim Policy: Retain (keep disk after PVC delete), Delete (destroy disk). Always use Retain for databases.

Chapter 6: Troubleshooting (30% of exam!)

The Troubleshooting Flowchart

Pod not running? ├── Pending? → kubectl describe pod → check Events │ ├── Insufficient CPU/memory → scale nodes or reduce requests │ ├── No matching node (taints) → add toleration or remove taint │ └── PVC not bound → check StorageClass, PV availability ├── CrashLoopBackOff? → kubectl logs --previous │ ├── Exit 137 → OOMKilled → increase memory limit │ ├── Exit 1 → app error → check logs for stack trace │ └── Exit 0 → app exits normally → check entrypoint/CMD ├── ImagePullBackOff? → wrong image name, private registry auth └── Running but not ready? → readiness probe failing └── kubectl describe → check probe config, app startup time

Essential Troubleshooting Commands

# Application debugging kubectl describe pod <name> -n <ns> # Events, conditions, status kubectl logs <pod> -n <ns> --previous # Crash logs from previous container kubectl logs <pod> -n <ns> -c <container> # Specific container in multi-container pod kubectl exec -it <pod> -- /bin/sh # Get a shell inside the container kubectl get events --sort-by='.lastTimestamp' -A # Cluster-wide events timeline # Node debugging kubectl describe node <name> # Conditions, capacity, allocations kubectl top nodes # Actual CPU/memory usage kubectl top pods -n <ns> --sort-by=memory # Find resource hogs kubectl cordon <node> # Prevent new scheduling kubectl drain <node> --ignore-daemonsets # Evict pods for maintenance # Network debugging kubectl run debug --rm -it --image=busybox -- nslookup <service> kubectl get endpoints <service> -n <ns> # Empty = no backend pods kubectl get svc -n <ns> # Check service selector + ports # Control plane debugging kubectl get componentstatuses # (deprecated but still works) kubectl get pods -n kube-system # Check control plane pods crictl ps -a # Container runtime level journalctl -u kubelet -f # Kubelet logs on node
Exam tip: Troubleshooting is 30% of CKA. The pattern is always: describe → logs → events → exec. For node issues: kubelet journal + crictl. For networking: run a debug pod + check endpoints. Practice this flow until it's muscle memory.

Chapter 1: Container Images & Multi-Container Pods

Dockerfile Best Practices

# Multi-stage build: build in one stage, run in minimal image FROM golang:1.22 AS builder WORKDIR /app COPY . . RUN CGO_ENABLED=0 go build -o server . FROM gcr.io/distroless/static:nonroot COPY --from=builder /app/server /server USER 65534:65534 ENTRYPOINT ["/server"] # Final image: ~15MB, no shell, non-root, minimal attack surface

Multi-Container Pod Patterns

  • Sidecar: Helper container that enhances the main container. Example: Envoy proxy for mTLS, Fluentd for log shipping.
  • Init Container: Runs before the main container starts. Use for: waiting for a database, downloading config, running migrations.
  • Ambassador: Proxy container that simplifies external access. Your app talks to localhost; the ambassador handles service discovery.
  • Adapter: Transforms output of the main container. Example: converts log format from proprietary to standard JSON.
# Init container: wait for database before app starts spec: initContainers: - name: wait-for-db image: busybox command: ['sh', '-c', 'until nc -z postgres-svc 5432; do echo waiting; sleep 2; done'] containers: - name: app image: my-app:v2
In your org: Your payment service pod has: init container (waits for Redis), main container (the app), sidecar (Envoy for mTLS). The init container ensures the app never starts without its dependency. The sidecar handles encryption without your app knowing.

Chapter 2: Configuration & Security

Service Accounts

Every pod runs as a service account. Service accounts provide identity for processes running in pods. By default, pods use the "default" service account, which may have too many permissions.

# Create dedicated service account kubectl create sa app-sa -n production # Disable auto-mounting of token (security best practice) apiVersion: v1 kind: ServiceAccount metadata: name: app-sa automountServiceAccountToken: false # Use in pod spec spec: serviceAccountName: app-sa

SecurityContext

# Pod-level security context spec: securityContext: runAsUser: 1000 runAsGroup: 3000 fsGroup: 2000 runAsNonRoot: true containers: - name: app securityContext: allowPrivilegeEscalation: false readOnlyRootFilesystem: true capabilities: drop: ["ALL"]

Resource Quotas & Limit Ranges

# ResourceQuota: limit total resources in a namespace apiVersion: v1 kind: ResourceQuota metadata: name: team-quota namespace: team-a spec: hard: requests.cpu: "10" requests.memory: "20Gi" limits.cpu: "20" limits.memory: "40Gi" pods: "50" --- # LimitRange: default limits for pods that don't specify them apiVersion: v1 kind: LimitRange metadata: name: default-limits namespace: team-a spec: limits: - default: cpu: "500m" memory: "256Mi" defaultRequest: cpu: "100m" memory: "128Mi" type: Container
In your org: Each team gets a namespace with a ResourceQuota. Team A gets 20 CPUs max. LimitRanges ensure that even if a developer forgets to set limits, their pod gets defaults instead of consuming unlimited resources.

Chapter 3: Deployment Strategies & Helm

Blue/Green Deployment

Run two identical environments. Blue = current production. Green = new version. Test green, then switch the Service selector from blue to green. Instant rollback = switch back to blue.

# Service currently pointing to blue spec: selector: app: my-app version: blue # Change to "green" to switch # Deploy green alongside blue kubectl apply -f deployment-green.yaml # Test green pods directly # Switch traffic: update service selector kubectl patch service my-app -p '{"spec":{"selector":{"version":"green"}}}' # Rollback: switch back to blue kubectl patch service my-app -p '{"spec":{"selector":{"version":"blue"}}}'

Canary Deployment

Route a small percentage of traffic to the new version. Monitor errors and latency. If healthy, gradually increase. If broken, route all traffic back to stable.

Helm Deep Dive

# Chart structure my-chart/ ├── Chart.yaml # Chart metadata (name, version) ├── values.yaml # Default configuration values ├── templates/ # K8s manifests with Go templates │ ├── deployment.yaml │ ├── service.yaml │ └── ingress.yaml └── charts/ # Dependency charts # Install with custom values helm install my-release ./my-chart \ --namespace production \ --values prod-values.yaml \ --set image.tag=v2.3.1 # List releases helm list -n production # Rollback to previous version helm rollback my-release 1 -n production # Template locally (dry-run to see generated YAML) helm template my-release ./my-chart --values prod-values.yaml

Chapter 4: Probes & Observability

Three Types of Probes

Liveness Probe: "Is the container alive?" If it fails, kubelet restarts the container. Use when your app can hang or deadlock.

Readiness Probe: "Is the container ready to serve traffic?" If it fails, the pod is removed from Service endpoints. Use for warm-up time.

Startup Probe: "Has the container started?" Disables liveness/readiness until it succeeds. Use for slow-starting apps (Java apps with 60s startup).

containers: - name: app livenessProbe: httpGet: path: /healthz port: 8080 initialDelaySeconds: 15 periodSeconds: 10 failureThreshold: 3 # 3 failures = restart readinessProbe: httpGet: path: /ready port: 8080 initialDelaySeconds: 5 periodSeconds: 5 startupProbe: httpGet: path: /healthz port: 8080 failureThreshold: 30 # 30 x 10s = 5 min max startup periodSeconds: 10

API Deprecations

K8s deprecates APIs and removes them after 2-3 versions. Check before upgrading:

# Check for deprecated APIs in your manifests kubectl convert -f old-manifest.yaml # (if available) # Or use kubent (kube-no-trouble) kubent # scans cluster for deprecated API versions # Common migrations: # extensions/v1beta1/Ingress → networking.k8s.io/v1/Ingress # policy/v1beta1/PodSecurityPolicy → replaced by Pod Security Standards

Chapter 5: CRDs & Extending Kubernetes

Custom Resource Definitions (CRDs)

Extend the Kubernetes API with your own resource types. Once defined, you can kubectl get them just like pods or services.

# CRD: define a new "DatabaseBackup" resource apiVersion: apiextensions.k8s.io/v1 kind: CustomResourceDefinition metadata: name: databasebackups.mycompany.com spec: group: mycompany.com versions: - name: v1 served: true storage: true schema: openAPIV3Schema: type: object properties: spec: type: object properties: database: type: string schedule: type: string scope: Namespaced names: plural: databasebackups singular: databasebackup kind: DatabaseBackup shortNames: [dbb] --- # Now you can create instances: apiVersion: mycompany.com/v1 kind: DatabaseBackup metadata: name: nightly-backup spec: database: production-postgres schedule: "0 2 * * *" # And query them: # kubectl get databasebackups # kubectl get dbb
In your org: CRDs + a custom controller (operator) = automation. You define a PostgresCluster CRD. The operator watches for it and automatically provisions a 3-node Postgres cluster with replication, backups, and failover. Your developers just write 10 lines of YAML instead of managing databases manually.

Chapter 3: System Hardening

AppArmor & Seccomp

AppArmor: Linux security module that restricts what a process can access (files, network, capabilities). Apply profiles to containers to limit their blast radius.

Seccomp: Filters system calls a container can make. Block dangerous syscalls like ptrace, mount, reboot.

# Seccomp profile in pod spec (K8s 1.19+) securityContext: seccompProfile: type: RuntimeDefault # Docker/containerd default profile # Or: type: Localhost # localhostProfile: profiles/my-profile.json # AppArmor annotation metadata: annotations: container.apparmor.security.beta.kubernetes.io/app: runtime/default

Minimize Host OS Attack Surface

  • Remove unnecessary packages from nodes
  • Disable SSH to nodes (use kubectl exec or node-shell instead)
  • Use immutable OS images (Bottlerocket, Flatcar, Talos)
  • Restrict IAM roles: nodes should have minimal cloud permissions

Secrets Encryption at Rest

# /etc/kubernetes/encryption.yaml apiVersion: apiserver.config.k8s.io/v1 kind: EncryptionConfiguration resources: - resources: - secrets providers: - aescbc: keys: - name: key1 secret: - identity: {} # fallback: unencrypted (for reading old secrets) # Apply: add --encryption-provider-config to API server manifest # Verify: etcdctl get /registry/secrets/default/my-secret # Should be encrypted, not readable

Runtime Sandboxing: gVisor & Kata

gVisor (runsc): User-space kernel that intercepts syscalls. Containers run in a sandbox — even if the app is compromised, it can't access the real kernel. Performance cost: ~10-30%.

Kata Containers: Lightweight VMs for each container. Stronger isolation than gVisor but heavier. Use for multi-tenant environments.

# RuntimeClass for gVisor apiVersion: node.k8s.io/v1 kind: RuntimeClass metadata: name: gvisor handler: runsc # Use in pod spec: runtimeClassName: gvisor

Chapter 6: Runtime Security & Monitoring

Falco: Runtime Threat Detection

Falco watches kernel syscalls in real time and fires alerts on suspicious activity: shell spawned in container, sensitive file read, unexpected network connection.

# Falco rule: alert when shell is opened in a container - rule: Terminal shell in container desc: A shell was spawned inside a container condition: > spawned_process and container and proc.name in (bash, sh, zsh) output: > Shell opened (user=%user.name container=%container.name image=%container.image.repository command=%proc.cmdline) priority: WARNING # Falco rule: detect crypto mining - rule: Detect crypto miners desc: Detect known crypto mining processes condition: > spawned_process and container and proc.name in (xmrig, minerd, cpuminer) output: "Crypto miner detected (container=%container.name)" priority: CRITICAL

# Find who deleted a deployment cat /var/log/k8s-audit.log | jq 'select(.verb=="delete" and .objectRef.resource=="deployments")' # Find all requests from unknown service accounts cat /var/log/k8s-audit.log | jq 'select(.user.username | startswith("system:serviceaccount") | not)'
